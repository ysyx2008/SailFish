# 主流大模型服务数据安全与隐私政策调研报告

> 调研日期：2026 年 3 月 13 日

## 摘要

本报告调研了国际与国内主流大语言模型（LLM）服务商的数据安全政策、用户协议关键条款，以及中国、欧盟、美国三大司法管辖区的法律监管要求。核心结论：

- **付费 API 调用总体安全**，但服务商之间政策差异显著
- **国际服务商**（OpenAI、Anthropic、Google）的 API 默认不将用户数据用于模型训练，且有明确的合同承诺
- **国内服务商**多数默认可能将去标识化数据用于训练，部分提供 opt-out 机制，但政策透明度参差不齐
- **中国法律明确要求**：将用户数据用于 AI 训练必须取得用户同意，且须提供便捷的撤回方式

## 一、国际服务商

### 1.1 OpenAI（GPT-4o / o3 / o4-mini 等）

| 项目 | 政策详情 |
|---|---|
| API 数据用于训练 | **默认不用**（自 2023 年 3 月 1 日起） |
| Opt-in 机制 | 仅通过 API 控制台主动勾选同意后才用于训练 |
| 数据保留 | 滥用监控日志保留 **30 天**，之后删除 |
| 高级选项 | Modified Abuse Monitoring（不记录内容）、**Zero Data Retention**（零保留） |
| 加密 | AES-256 静态加密 + TLS 1.2+ 传输加密 |
| 数据所有权 | 用户完全拥有输入和输出 |
| 企业级 | 支持自带加密密钥（Enterprise Key Management） |

**政策来源**：[Enterprise privacy at OpenAI](https://openai.com/policies/api-data-usage-policies)

**评价**：政策最透明、选项最丰富。企业客户可实现真正的零数据保留。免费 Playground 和 ChatGPT 消费端政策不同，使用时需注意区分。

---

### 1.2 Anthropic（Claude 3.5 / Claude 4 等）

| 项目 | 政策详情 |
|---|---|
| API 数据用于训练 | **默认不用** |
| Opt-in 机制 | 仅通过 Developer Partner Program 主动加入才用于训练 |
| 数据保留 | API 输入/输出 **30 天内自动删除** |
| 零保留协议 | 支持签署零数据保留协议 |
| 合规文件 | 提供 DPA（数据处理协议）、Commercial Terms |
| 消费端区别 | Claude Free/Pro/Max 有不同政策，用户可选择是否允许训练 |

**政策来源**：[Anthropic Privacy Center](https://support.anthropic.com/en/articles/7996868)

**评价**：API 层面政策最干净——默认不训练、30 天自动删除、支持零保留协议。是目前对用户数据保护最明确的国际服务商。

---

### 1.3 Google（Gemini API）

| 项目 | 政策详情 |
|---|---|
| 付费 API 数据用于训练 | **不用** |
| 免费层 | **可能用于训练**（Starter Edition 允许） |
| 数据保留 | 滥用检测日志保留 **55 天** |
| 审核机制 | 自动化 + 人工审核（仅授权员工通过内部治理平台访问） |
| 企业版 | Cloud Data Processing Addendum 保障 |
| Gemini for Google Cloud | 明确不用于训练模型 |

**政策来源**：[Gemini API Usage Policies](https://ai.google.dev/gemini-api/docs/usage-policies)

**评价**：付费 API 安全可靠，但**免费层有训练风险**。日志保留 55 天较 OpenAI/Anthropic 的 30 天更长。

## 二、国内服务商

### 2.1 DeepSeek

| 项目 | 政策详情 |
|---|---|
| 数据用于训练 | **默认可能用于训练**（去标识化后） |
| Opt-out | 可关闭"数据用于优化体验"开关 |
| 数据处理 | 承诺安全加密 + 去标识化，声称无法重新识别个人 |
| 特别提示 | 隐私政策明确建议用户**不要主动输入个人敏感信息** |
| 开发者责任 | API 开发者需自行承担数据处理责任，向终端用户披露信息处理规则 |

**政策来源**：[DeepSeek 隐私政策](https://cdn.deepseek.com/policies/zh-CN/deepseek-privacy-policy.html)（2026 年 2 月 10 日更新）

**评价**：提供 opt-out 是好的，但"默认用于训练"的立场与国际服务商相反。API 用途与消费端的政策边界不够清晰。

---

### 2.2 百度文心一言

| 项目 | 政策详情 |
|---|---|
| 数据用于训练 | **去标识化后可用于训练**（提高理解能力和响应质量） |
| Opt-out | 记忆簿内容需事先许可才用于训练；普通对话的 opt-out 机制不明确 |
| 数据保留 | "在法律允许的时间内保留" |
| 脱敏方式 | 内容替换、匿名处理、去标识化 |
| 用户权利 | 可查阅/删除问答历史，可管理记忆簿 |

**政策来源**：[文心一言个人信息保护规则](https://yiyan.baidu.com/docUrl/EB118_infoprotect/llf9tqa4r)

**评价**：对普通对话数据的训练使用采取默认许可态度，opt-out 机制不够便捷和明确。数据保留期限表述模糊。

---

### 2.3 字节豆包

| 项目 | 政策详情 |
|---|---|
| 数据用于训练 | **去标识化后可能用于训练**（优化模型与产品体验） |
| Opt-out | 可在设置中关闭"帮助模型改进效果"开关 |
| 数据原则 | 最小必要原则、按需授权、分场景告知 |
| 敏感权限 | 默认不开启（麦克风、相机等） |

**政策来源**：[豆包隐私政策](https://www.doubao.com/legal/privacy)（2026 年 2 月 10 日更新）

**评价**：提供了明确的 opt-out 开关，政策表述相对清晰。但"去标识化后可用于训练"仍是默认立场。

### 2.4 阿里通义千问

| 项目 | 政策详情 |
|---|---|
| 数据用于训练 | **协议中未直接明确说明** |
| 安全措施 | TLS 传输加密、访问控制、DDoS 防护、WAF、实时监控 |
| 隐私标准 | 声称遵守 GDPR 和 CCPA，采用匿名化处理 |
| 服务定位 | 用户协议偏向个人学习、研究、日常使用场景 |

**政策来源**：[千问用户服务协议](https://terms.alicdn.com/legal-agreement/terms/c_end_product_protocol/20231011201348415/20231011201348415.html)

**评价**：安全技术措施较完善，但在"是否用于训练"这一关键问题上缺乏明确表态，用户难以做出知情判断。

### 2.5 智谱 GLM

| 项目 | 政策详情 |
|---|---|
| 数据用于训练 | **协议中未明确说明** |
| 数据所有权 | 明确表示用户完全拥有自己的数据 |
| 数据处理 | 最小必要原则、匿名化 + 加密 |

**评价**：强调数据所有权归用户，态度保守，但在训练使用问题上缺乏明确的肯定或否定。

## 四、法律监管环境

### 4.1 中国

中国是全球最早对生成式 AI 进行专门立法的国家之一，形成了多层次法律框架：

**《生成式人工智能服务管理暂行办法》（2023 年 8 月 15 日施行）**

- 使用个人信息训练模型**必须取得个人同意**
- 不得收集非必要的个人信息
- 不得非法留存能够识别用户身份的输入信息和使用记录
- 不得非法向他人提供用户的输入信息和使用记录
- 主管部门可要求说明训练数据来源、规模、类型等信息

**《个人信息保护法》（2021 年 11 月 1 日施行）**

- 第 13 条：处理个人信息应取得个人同意
- 第 14 条：同意必须在**充分知情的前提下自愿、明确作出**
- 第 15 条：用户有权**撤回同意**，企业应提供便捷的撤回方式
- 禁止"捆绑同意"——用户应能在不授权训练的情况下使用核心服务功能
- 禁止"一揽子同意"——基础服务、模型训练、个性化推荐等不同目的应分别获取同意

**现实合规状况**

据南都 2024 年 12 月测评报告，国内主流平台普遍存在以下问题：
- 多数未明确告知用户数据是否用于 AI 训练
- 未提供便捷的拒绝和退出渠道
- 超四成受访者不赞成个人数据被用于训练

这意味着部分国内服务商的实际做法可能与法律要求存在差距。

---

### 4.2 欧盟

**GDPR（《通用数据保护条例》）**

- **数据最小化**（第 5(1)(c) 条）：只能处理特定目的所必需的数据
- **目的限制**（第 5(1)(b) 条）：为一个目的收集的数据不能未经同意挪作训练用途
- **解释权**（第 22 条）：用户有权不受纯自动化决策的约束，且有权获得 AI 决策的解释
- 数据一旦进入模型权重，删除实际上不可执行——因此预防是唯一可行的合规策略
- 截至 2024 年，GDPR 累计罚款已达 **58.8 亿欧元**

**《欧盟人工智能法案》（EU AI Act）**

- 通用 AI 模型条款于 2025 年 8 月 2 日生效
- 要求公开训练数据摘要（六类数据源：公开数据集、授权数据、爬取内容、用户数据、合成数据、其他）
- 多数条款于 **2026 年 8 月 2 日** 全面生效
- 要求粒度化的同意获取、按目的的 opt-in、便捷的 opt-out

---

### 4.3 美国

**联邦层面**

美国目前缺乏综合性联邦 AI 法案。现政府倾向于减少监管，已撤销前任关于 AI 安全的行政命令。联邦层面仅有针对深度伪造等单一领域的法案（如 TAKE IT DOWN Act）。

**州级监管（各州差异大）**

- **加州（最严格）**：
  - SB 53《前沿 AI 透明度法案》：大模型开发者须发布风险框架，违规最高罚 100 万美元
  - AB 2013《AI 训练数据透明度法案》：必须公开训练数据源、数据类型及版权信息
- **科罗拉多州**：首部综合性州级 AI 法（2026 年 6 月 30 日生效），对开发者和使用者均有约束
- 超过 30 个州已制定或正在考虑 AI 相关立法
- 2025 年全美超 1000 项 AI 法案被引入各州议会

## 五、对 SailFish 用户的建议

### 5.1 风险评估

SailFish 作为 AI Agent 桌面应用，发送给 AI 服务的数据可能包含：
- 终端输出（含 IP 地址、文件路径、进程信息）
- 命令历史（含可能的密码、密钥、Token）
- 服务器配置信息（SSH 连接参数、主机名）
- 业务代码和日志内容

这些信息的敏感程度远高于普通聊天对话。

### 5.2 服务商选择建议

| 安全需求 | 推荐方案 |
|---|---|
| **最高安全**（金融、政务、军工） | 本地部署开源模型（DeepSeek、Qwen 等） |
| **高安全**（企业内部、生产服务器） | Anthropic API / OpenAI API（付费，默认不训练） |
| **一般安全**（个人开发、学习） | 上述国际 API 或国内 API（确保关闭训练开关） |
| **低敏感**（公开信息处理） | 任意服务商均可 |

### 5.3 使用建议

1. **优先选择 API 而非消费端产品**：API 的数据保护政策普遍优于 ChatGPT、Claude.ai 等消费端
2. **国内服务务必检查设置**：确认关闭"数据用于模型改进"类开关
3. **避免在 prompt 中包含明文密钥**：即使服务商承诺不训练，数据仍会短暂经过其服务器
4. **敏感环境考虑本地模型**：SailFish 未来可支持本地模型接入，对于高敏感场景是最安全的选择
5. **定期审查服务商政策更新**：隐私政策可能随时间变化，建议每半年复查一次
